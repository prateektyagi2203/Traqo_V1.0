# 🎯 RAG Transparency Dashboards - Quick Start Guide

## Problem Solved

Your RAG feedback system was a **complete black box**. You couldn't see:
- Which patterns the system trusts (and penalties applied)
- Why signals were being blocked
- Which patterns actually work vs don't work
- What happens if you adjust thresholds

**Now you can see everything.** Here's how to get started:

---

## Step 1: Extract Dashboard Data (DO THIS FIRST!)

Run this command in the terminal:

```bash
python _extract_dashboard_data.py
```

**What it does:**
- Reads your feedback system and extracts pattern status
- Analyzes closed trades to calculate performance metrics
- Generates 3 JSON files that dashboards read from
- Takes 10-15 seconds

**Output:**
```
✅ DATA EXTRACTION COMPLETE

Generated files:
  1. paper_trades/pattern_status.json
  2. paper_trades/trade_performance.json
  3. paper_trades/audit_status.json
```

---

## Step 2: Open the Dashboards (In This Order!)

### **Dashboard 1: RAG Rules Explorer** ⭐ START HERE
**File:** `rag_rules_explorer.html`

**What you'll see:**
- Every pattern your system knows about
- Which ones are **PENALIZED** (blocked) ❌
- Which ones are **BOOSTED** (encouraged) ✓
- Which are **WHITELISTED** (production patterns) 🎯
- Win rates and trade counts per pattern

**Key insight:** You'll immediately see that **all 7 whitelisted patterns are penalized** due to low actual win rates (10-25% vs 45% threshold). This explains why NO signals are being generated.

**How to use:**
1. Open in any web browser
2. Search patterns by name
3. Filter by status (All / Penalized / Boosted / Whitelisted)
4. Check the stats cards at top for quick overview

---

### **Dashboard 2: Trade Performance** 
**File:** `trade_performance_dashboard.html`

**What you'll see:**
- Actual performance of each pattern from closed trades
- Win rate, profit factor, wins/losses for each pattern
- Sort by profit factor, win rate, or trade count
- Comparison of live performance vs what RAG expects

**Key insight:** Validates that RAG penalties are justified. Patterns the system blocked ACTUALLY DON'T WORK (in current market).

**How to use:**
1. Sort by "Profit Factor" to see best/worst performers
2. Check if any penalized patterns actually perform well
3. Identify which patterns have enough trades for statistical confidence

---

### **Dashboard 3: Threshold Simulator** 
**File:** `threshold_simulator.html`

**What you'll see:**
- Interactive sliders for Win Rate, Profit Factor, Trade Count thresholds
- Real-time list of patterns that would qualify at different settings
- Forecast impact before making changes

**Key insight:** If you lower WR threshold from 45% to 20%, which patterns become tradeable?

**How to use:**
1. Adjust Win Rate slider (left side)
2. See patterns light up green when they would qualify
3. Test different threshold combinations
4. Note the settings you want to try

---

### **Dashboard 4: Scan Audit Viewer** 
**File:** `scan_audit_viewer.html`

⚠️ **This one requires running the trading system first!**

**What you'll see:**
- Every signal generated by the scanner
- For each signal: whether it was ENTERED or REJECTED
- Detailed breakdown of rejection reasons (which filters blocked it)
- Filter by date, ticker, or decision type

**How to enable it:**
1. Run: `python paper_trader.py` (on next trading day)
2. This generates audit logs
3. Come back to this dashboard to see signal filtering in real-time

## Next Steps (Based on Your Findings)

### Option A: Lower Win Rate Threshold (Quick Fix)
- **What:** Reduce MIN_WIN_RATE from 45% to 20%
- **Effect:** 3-4 patterns become tradeable immediately
- **Risk:** May accept poor performers
- **Timeline:** 1 minute change + test 1-2 weeks
- **Use:** Threshold Simulator to see impact first

### Option B: Regime-Aware RAG (Better Fix)
- **What:** Separate pattern performance by market condition (bullish/bearish/neutral)
- **Effect:** Patterns might work in specific conditions
- **Risk:** Lower - only trades in favorable conditions
- **Timeline:** 2-3 hours implementation
- **Use:** Requires code changes to statistical_predictor.py

### Option C: Expand Whitelist (Experimental)
- **What:** Add 3-5 patterns from excluded list
- **Effect:** More signals, but untested patterns
- **Risk:** High - added patterns were excluded for reason
- **Timeline:** 5 minutes
- **Use:** Check Trade Performance Dashboard first

### Option D: Data Investigation (Institutional)
- **What:** Compare backtest data (2016-2023) vs live data (2024-2026)
- **Effect:** Understand if patterns genuinely degraded or backtest was wrong
- **Risk:** None - data only
- **Timeline:** 1-2 hours
- **Use:** Needed before any threshold changes

## Troubleshooting

### Dashboard shows "No Data Available"
**Fix:** Run the extraction script:
```bash
python _extract_dashboard_data.py
```

### Scan Audit Viewer shows no logs
**Fix:** Need to run the paper trader:
```bash
python paper_trader.py
```
This generates audit logs for the next trading day (March 11).

### Data looks old/stale
**Fix:** Re-run extraction:
```bash
python _extract_dashboard_data.py
```
This reloads from your feedback system.

## Technical Details

### Data Files Generated
- `pattern_status.json` - Pattern penalties/boosts from feedback system
- `trade_performance.json` - Closed trade analysis by pattern
- `audit_status.json` - Metadata about available audit logs
- `audit_decisions_YYYY-MM-DD.jsonl` - Generated by paper_trader.py

### Safe to Open In
- Any modern web browser (Chrome, Firefox, Safari, Edge)
- Can be opened offline (dashboards work locally)
- No data is sent anywhere

### How Data Updates
```
paper_trades.db
    ↓
feedback/learned_rules.json
    ↓
_extract_dashboard_data.py
    ↓
pattern_status.json ← Used by Dashboards
trade_performance.json ← Used by Dashboards
```
